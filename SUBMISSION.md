# Veritas — Submission

**Stellar Hacks: Real-World ZK** · Privacy-Preserving Credential Verifier on Stellar

> **Prove you're compliant — without revealing the document.** Veritas is a ZK
> credential-compliance layer for Soroban: holders generate Groth16 proofs of
> *derived predicates* (18+, jurisdiction-allowed, valid-owner), and a Soroban
> contract verifies them **on-chain** via a real BN254 pairing, gating a
> compliant action while the underlying DOB / country / identity stay private.

---

## The 60-second pitch

Compliance is the **#1 barrier** to real-world on-chain payments. Today, proving
you may transact means uploading a passport to a centralized oracle — a privacy
and custody nightmare. Veritas replaces document disclosure with **cryptographic
proof of a derived predicate**: the verifier learns *only* "this person is 18+",
never their date of birth. One architecture lands three of the hackathon's listed
use-cases simultaneously — **identity verification without document exposure**,
**private payments**, and **confidential tokens**.

## What makes it unique

1. **Real BN254 pairing inside Soroban WASM.** Stellar has no BN254 precompile,
   so the heart of this submission is a Groth16 verifier that performs the full
   miller-loop + final-exponentiation **in the contract itself** (via
   `ark-bn254`, `#![no_std]`). It compiles to **~73 KB** and verifies genuine
   snarkjs proofs — not a stub, not a fake "verified" flag.
2. **Compliance-as-a-private-predicate on Stellar** is a fresh framing: the same
   issuer-signed credential yields age, jurisdiction, and ownership proofs,
   reusable across dApps, with **on-chain double-spend protection** via per-use
   nullifiers.
3. **End-to-end reproducibility.** One script (`demo.sh`) takes you from signed
   credential → Groth16 proof → on-chain verification → gated faucet unlock →
   tampered-proof rejection. Every byte the contract sees is generated by the
   open pipeline.

## The ZK story

- **Issuer** signs a credential claim with **EdDSA-Poseidon over BabyJubJub**.
- **Prover** (off-chain) computes a **Groth16** proof that a *derived predicate*
  holds. An invalid predicate (underage, wrong country, forged signature) makes
  the circuit **unsatisfiable** — no proof can ever exist.
- **Soroban verifier** runs the BN254 pairing check on-chain, enforces that the
  issuer pubkey and allow-set Merkle root match on-chain values, records a
  nullifier, and unlocks the action. The proof reveals **nothing** about the
  underlying attribute.

Circuits: `AgeGte`, `ValidOwner`, `JurisdictionAllowed`, and the flagship
`CredentialAge` (issuer-signed credential whose private `age ≥ threshold`,
emitting a nullifier).

## Demo (2–3 min)

```bash
bash scripts/demo.sh
```
Watch, in order:
1. Issuer signs demo credentials (BabyJub EdDSA-Poseidon).
2. Prover generates Groth16 proofs; snarkjs verifies them off-chain (`OK!`).
3. Proofs are encoded and verified **on-chain** by the Soroban contract —
   `AgeGte` unlocks the faucet, `ValidOwner` enforces double-spend protection,
   `JurisdictionAllowed` binds to the posted Merkle root.
4. A **tampered** credential (threshold bumped 18→21) and a **cross-circuit**
   proof are both **rejected**.

Then deploy + submit a real proof on testnet:
```bash
export STELLAR_ADMIN_SECRET="<funded testnet secret>"
bash scripts/deploy_testnet.sh   # verify_for(...) -> true  on Stellar testnet
```

## Real-world relevance

KYC/AML/sanctions compliance is the gate every regulated payment passes through.
Veritas turns that gate into a privacy-preserving on-chain primitive: a stablecoin
issuer can gate transfers by jurisdiction, an exchange can prove accreditation
without PII, a DeFi protocol can enforce sanctions-screening via an on-chain
allow-set root. That is the exact wedge banks cite as the reason they can't settle
on public chains today.

## Meaningful ZK integration with Stellar

- The verifier **is** the submission's core: a deployed Soroban contract that
  verifies real Groth16 proofs by computing BN254 pairings in WASM.
- Correctness is proven two ways: (a) the shared pairing core is unit-tested
  against real snarkjs proofs; (b) a Soroban sandbox test deploys the contract
  and verifies the same proofs **on-chain** (6/6 tests pass).
- The deploy script submits the identical proof bytes to **Stellar testnet**.

## Scope & stretch

**MVP (done):** 4 circom circuits with real proofs · on-chain BN254 Groth16
verifier · issuer registry, nullifier set, allow-set root · gated faucet action ·
off-chain issuer + prover · sandbox + testnet demo · 6 passing tests.

**Stretch (described, not built):** OFAC-style sanctions **non-membership**
proofs (proving "not in" a posted Merkle root) and a compliance-gated
**confidential token transfer**. The verifier API and circuit structure already
support adding these without redesign.

## Trusted setup & honesty

We use the standard public powers-of-tau with a single contribution
(**hackathon-grade**; production needs a multi-party ceremony). Demo issuer keys
are generated fresh and marked `DO NOT USE IN PROD`. On-chain pairing gas is
heavier than a hypothetical native precompile — documented plainly rather than
hidden.
